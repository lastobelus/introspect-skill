# External Review Mechanics

Use real external reviews when the active environment provides a reviewer mechanism
such as Codex subagents, `codex exec -s read-only`, or `opencode run`. Do not simulate
reviewer output.

This repo does not depend on a project-specific reviewer router. If a future branch
adds one, follow that repo guidance. Otherwise use the simple selection below.

## Reviewer Selection

1. Prefer a product-native subagent/reviewer tool when it is available.
2. Otherwise use `codex exec` in read-only mode from the repo root when installed.
3. Otherwise use `opencode run` from the repo root when installed.
4. If no eligible reviewer can run after one diagnosis/retry pass, report the blocker.

Keep reviewers read-only. They may inspect files, diffs, plans, and allowed docs. They
must not run validation, tests, installs, migrations, servers, browser automation, or
commands that read protected local stores unless the owning skill explicitly permits a
sanitized local summary.

## Protected Context

Scan the plan, diff, context list, and prompt for protected sources:

- `*.env`, keys, tokens, auth stores, browser profiles, local tool-state blobs
- raw `~/.t3`, `~/.claude`, `~/.codex`, or opencode logs/databases
- full provider logs or native session logs that may contain private prompts

Do not point reviewers at protected raw files. Provide synthetic fixtures, sanitized
excerpts, or summaries generated by this repo's scripts instead.

## Prompt Shape

Build a bounded prompt in memory. Include:

- task type: plan review, implementation review, or PR review
- current source-of-truth file or PR/base/head refs
- focused context file list
- touched file list or short diff summary
- validation already run, if any
- active lens from `repo-lenses.md`
- finding cap for the current round

Ask for findings first, ordered by severity. Require distinct, actionable findings and
ask the reviewer to state when the lens is clean. JSON is preferred when the caller has
a parser; otherwise accept concise Markdown and normalize findings in the main session.

## Session Handling

Record when a reviewer starts and keep the session id or log path visible when
available. External reviews can take several minutes. Do not declare a review stuck
solely because time has passed; inspect for an approval wait, auth/session prompt,
explicit error, crashed process, or continued progress before retrying.

Retry once with the same reviewer only when invocation or extraction fails for a
fixable reason. If a reviewer ran forbidden validation commands, ignore command-derived
evidence, salvage file/docs-based findings, and tighten the next prompt.
